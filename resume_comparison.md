# Resume Comparison

This document conducts a harsh, realistic evaluation of the difference between the prior repo templates/concepts and the newly reconstructed Master Resumes.

---

## 1. Technical Support Engineer (TSE) Lane

### What Changed?
* **What got stronger:** The framing of the ticket workflow. The new version strips out vague descriptions and clarifies that Isaiah processes 2-10 tickets a shift, doing fault isolation before escalating. It is grittier and more real.
* **What got more credible:** The projects section. Removed all subtle phrasing that might have implied Live Pilot or Impulse were professional employer projects. They are now strictly defined as "Beta" and "Functional Prototype".
* **What got easier to scan:** The bullet length. Fluff has been burned out. 
* **What got removed:** Duplicate phrasing from administrative roles. "Implemented company policies" is gone. 
* **What is still missing from the candidate profile:** Enterprise-level monitoring tools (Zabbix, Datadog), active directory management, or deeper networking (CCNA-level routing/switching). The candidate just doesn't have it.

### Evaluation Scores (Old vs. New)

| Metric | Old Repo Version | New Firewall Version |
|--------|------------------|----------------------|
| **First-scan clarity** | 7/10 | 9/10 |
| **Credibility** | 8/10 | 10/10 |
| **Role alignment** | 8/10 | 9/10 |
| **Bullet sharpness** | 6/10 | 9/10 |
| **Interview defensibility** | 7/10 | 10/10 |
| **Overall conversion likelihood** | 7/10 | 8.5/10 |

*Harsh Truth:* The new TSE resume is highly defensible, but the ceiling for conversion remains tied to the fact that the candidate's core networking depth is relatively shallow. It converts perfectly for Tier 1/2 roles, but will still be rejected for complex Tier 3 infrastructure roles despite the rewrite.

## 2. Solutions Engineer / Implementation Bridge Lane

### What Changed?
* **What got stronger:** Translating side-projects into architectural language. Pointing out the shift from 30+ iterative LLM calls to 1 deterministic pipeline creates an immediate interview hook.
* **What got more credible:** Emphasizing the *bridge*. The old versions sometimes tried to hide the support nature of the Nexus WiFi role. The new version explicitly leans into it as proof of user empathy, framing it as: "I build complex systems, but I also have 1,500 hours explaining technology to angry users."
* **What got easier to scan:** Shifting the technical skills to highlight actual stack components (Python, React, API integration) directly beneath the summary to bypass the "Support Tech" title bias.
* **What got removed:** Any implication that he has operated as a Solutions Engineer in an enterprise B2B sales cycle. We removed the "Solutions focused" fluff from the summary.
* **What is still missing from the candidate profile:** Real B2B pre-sales, RFP responses, or enterprise software deployment experience. 

### Evaluation Scores (Old vs. New)

| Metric | Old Repo Version | New Firewall Version |
|--------|------------------|----------------------|
| **First-scan clarity** | 7/10 | 9/10 |
| **Credibility** | 6/10 | 9/10 |

| **Bullet sharpness** | 5/10 | 9/10 |
| **Interview defensibility** | 5/10 | 9/10 |
| **Overall conversion likelihood** | 5/10 | 7.5/10 |

*Harsh Truth:* The SE lane will inherently have a lower conversion rate than the TSE lane for Isaiah right now. You cannot fake B2B SaaS implementation experience. Overclaiming project scope in the old versions created severe interview risks (if they discovered the "app" was built in a bedroom). The new version fully owns the self-taught builder narrative, making it bulletproof in an interview, but it will still fail screens expecting formal SE history. Conversion relies on a hiring manager willing to take a chance on a hungry builder with great support chops.
