# Comparison v3

This document traces the evolution of the resume methodology across three phases:
- The **Previous Repo Resume** (prior historical iterations)
- The **Current Rebuilt Version** (v2, incorporating the initial Claims Firewall)
- The **Newly Corrected v3 Version** (aligning internal truth with external ATS and recruiter guidelines)

---

## Technical Support Engineer (TSE)

| Metric | Previous Repo -> v2 Rebuild | v2 Rebuild -> Corrected v3 Version |
|--------|-----------------------------|------------------------------------|
| **First-scan clarity** | **Much stronger**<br>Removed extreme amounts of boilerplate fluff (e.g., "implemented company procedures"). | **Stronger**<br>Shifted paragraphs in summaries to 2-3 sentence blocks, speeding up immediate read-times. |
| **Credibility** | **Much stronger**<br>Eliminated fake degree names and removed inflated "advanced enterprise implementation" phrasing. | **Slightly stronger**<br>Reduced numerical task estimates (like 2-10 tickets a shift) which can trigger skepticism, focusing instead on observable structural capacity. |
| **ATS safety** | **Stronger**<br>Organized the layout strictly around core sections. | **Much stronger**<br>Normalized all keyword use inline within bullets, preventing list-stuffing which ATS filters penalize. |
| **Keyword relevance** | **Unchanged**<br>Required keywords were present, but randomly placed. | **Much stronger**<br>Referred to specific hardware (APs, gateways) and network methodologies cleanly integrated into action bullets. |
| **Role alignment** | **Stronger**<br>Divided into lane-specific targets to stop trying to be an engineer and an admin simultaneously. | **Stronger**<br>The summary no longer focuses on the "objective" or what the candidate wants, but solely outlines the exact capabilities being offered to the hiring company. |
| **Interview defensibility** | **Much stronger**<br>Adherence to the truth firewall removed all hallucination risk. | **Stronger**<br>By avoiding exact SLA or daily completion rate estimations, there is zero risk of an interviewer cornering the candidate on metrics they can't mathematically prove. |

## Solutions Engineer / Implementation Bridge

| Metric | Previous Repo -> v2 Rebuild | v2 Rebuild -> Corrected v3 Version |
|--------|-----------------------------|------------------------------------|
| **First-scan clarity** | **Stronger**<br>Extracted project data and positioned them as strong proof points. | **Stronger**<br>Reduced the density of paragraphs and clarified the architectural terminology in the bullet points. |
| **Credibility** | **Much stronger**<br>Stopped positioning personal applications as enterprise startups with active user bases. | **Slightly stronger**<br>Explicit classification as "Functional Prototype" and "Public Beta" builds immediate trust with a technically literate hiring manager. |
| **ATS safety** | **Weaker**<br>The v2 rebuild broke standard formatting by putting "Projects" before "Experience" to hide the Support title. | **Much stronger**<br>v3 restores standard chronologies (Experience before Projects) ensuring standard ATS semantic parsers correctly log the document. |
| **Keyword relevance** | **Strongly stronger**<br>Proper definitions for Python tools, CUDA, and OSC protocols. | **Stronger**<br>Context-driven skill references make the integrations more robust. |
| **Role alignment** | **Stronger**<br>Created a dedicated 'bridge' narrative highlighting both technical building and customer support. | **Stronger**<br>Positions the candidate exactly as they are: a hybrid specialist lacking formal SE roots but carrying massive builder capability. |
| **Interview defensibility** | **Much stronger**<br>Candidate owns exactly what they did without over-staking claim on non-existent B2B enterprise sales pipelines. | **Unchanged**<br>The v2 constraints on truthfulness remain fundamentally solid in v3. |
