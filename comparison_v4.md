# Comparison v4

This document traces the evolution of the resume methodology across three phases:
- The **Previous Repo Resume** (prior historical iterations)
- The **Current Rebuilt Version** (v2, incorporating the initial Claims Firewall)
- The **Newly Corrected v4 Version** (strictly aligned to internal truth constraints and direct wording)

---

## Technical Support Engineer (TSE)

| Metric | Previous Repo -> v2 Rebuild | v2 Rebuild -> Corrected v4 Version |
|--------|-----------------------------|------------------------------------|
| **First-scan clarity** | **Much stronger**<br>Removed extreme amounts of boilerplate fluff. | **Stronger**<br>Shifted paragraphs in summaries to 2-3 sentence blocks, focusing entirely on direct experience. |
| **Credibility** | **Much stronger**<br>Eliminated fake degree names and removed out-of-scope implementation phrasing. | **Stronger**<br>Replaced task estimations with defined structural scopes, avoiding unverified numerical precision. Removed abstract corporate praise. |
| **Formatting consistency** | **Stronger**<br>Organized the layout strictly around core sections. | **Stronger**<br>Normalized tool usage inline within bullets, preventing isolated lists of keywords. |
| **Keyword relevance** | **Unchanged**<br>Required keywords were present, but randomly placed. | **Stronger**<br>References to specific hardware (APs, gateways) and network actions are integrated cleanly into action statements. |
| **Role alignment** | **Stronger**<br>Divided into lane-specific targets. | **Stronger**<br>The summary no longer outlines an "objective," but instead directly states the candidate's existing background. |
| **Interview defensibility** | **Much stronger**<br>Adherence to the truth firewall removed hallucination risk. | **Stronger**<br>By avoiding abstract compliance phrases and high-volume claims, the candidate's claims perfectly mirror their actual work. |

## Solutions Engineer / Implementation Bridge

| Metric | Previous Repo -> v2 Rebuild | v2 Rebuild -> Corrected v4 Version |
|--------|-----------------------------|------------------------------------|
| **First-scan clarity** | **Stronger**<br>Extracted project data and positioned them clearly. | **Stronger**<br>Reduced paragraph density and simplified technical terminology in the bullet points. |
| **Credibility** | **Much stronger**<br>Stopped positioning personal applications as enterprise startups. | **Stronger**<br>Explicit classification as "Functional Prototype" and explicit removal of phrases like "enterprise deployments" and "production". |
| **Formatting consistency** | **Weaker**<br>The v2 rebuild broke standard formatting by placing Projects before Experience. | **Stronger**<br>v4 restores standard chronology (Experience before Projects) ensuring standard section priorities are followed. |
| **Keyword relevance** | **Stronger**<br>Proper definitions for Python tools, CUDA, and OSC protocols. | **Stronger**<br>Maintained context-driven skill references with direct, unpolished language. |
| **Role alignment** | **Stronger**<br>Created a dedicated 'bridge' narrative. | **Stronger**<br>Positions the candidate exactly as they are: a technical support professional with independent coding experience. |
| **Interview defensibility** | **Much stronger**<br>Candidate owns exactly what they did without claiming B2B enterprise history. | **Stronger**<br>Constraints remain tight and all remaining vocabulary maps cleanly to verified facts in the master profile. |
